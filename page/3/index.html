Linux 系统编程 -- 内存映射

发表于 2015-08-02   |   分类于 系统编程

内存映射

　　进程可以将磁盘中的文件映射到其 virtual memory 中，这称为内存映射( Memory Mapping )。当文件被映射到 virtual memory 中，进程就可以访问内存的相应区域，以此来访问文件。

　　实际上，当文件映射到 virtual memory 时，文件并不存在于 physical memory 中。当进程需要访问文件时，相应的 page 才会从 virtual memory 换入到 physical memory 中。

　　另一方面，也可以将 virtual memory 的一个区域映射到匿名文件( Anonymous File )。当进程引用到这个区域的一个 virtual page 时，内核则会在 physical memory 中找到一个 victim page，如果这个 page 被修改过，则将这个 page 换出，接着将这个 page 的内容初始化为 0 并更新进程的 page table。

共享对象

　　我们知道，一个程序可以对应多个进程，那么，这些进程的会拥有相同的 .text 区域，另一方面，进程也会拥有相同的 shared library。假定每个进程都在 physical memory 中拥有这些相同的对象的一个副本，则无疑会占用过多的内存。

　　一个进程所映射的内存可以与其它进程共享，也就是说，同一个程序的多个实例会共享相同的 .text 区域，并且共享相同的 shared library，这些相同的对象在 physical memory 只存在一份副本。一个对象可以映射到多个进程的 virtual memory 的一个区域，可以分为共享映射和私有映射。

　　在共享映射的情况下，多个进程将一个对象映射到它们各自的 virtual memory 的一个区域，那么，要是有一个进程修改了这个对象，这个修改的操作对其它进程来说都是可见的。

　　私有映射则不同，私有映射采用了写时拷贝( Copy On Write )的技术。假定两个进程将一个对象映射到它们各自的 virtual memory 的一个区域：

　　physical memory 的一个区域在两个进程之间共享，要是有一个进程试图修改这个区域的一个 page，则会发生 exception，内核执行相应的 exception handler，在 physical memory 中创建这个 page 的一个拷贝，并更新进程的 page table，这时进程就可以放心地对这个新的 page 执行修改操作了。

fork 的语义

　　我们知道，调用fork()则会创建一个与父进程几乎一致的子进程，设想一下，在调用fork()时，要是系统完整地拷贝（即深度拷贝）父进程，那这代价也太大了。

　　实际上，调用fork()时，系统会为子进程创建必要的数据结构（ page table 或 file descriptor table 等等），子进程的 virtual memory 和父进程的 virtual memory 是一致的，当父进程或者子进程需要对共享区域进行修改时，则会使用 Copy On Write 的技术，创建需要修改的 page 的副本，并对这个副本执行修改操作。

创建映射

　　mmap()请求内核将文件的一个区域映射到进程的 virtual memory 的一个区域，通常将addr设置成NULL。

　　prot用来指定映射区域的访问权限：PROT_NONE 表示无法访问这个区域，PROT_READ 表示这个区域的 page 可读，PROT_WRITE 表示这个区域的 page 可写，PROT_EXEC 表示这个区域的 page 可被执行。

　　而flags则可以用来指定创建的是共享映射还是私有映射，是文件映射还是匿名映射：MAP_PRIVATE 表示创建的是私有映射，MAP_SHARED 表示创建的是共有映射，MAP_ANONYMOUS 表示创建的是匿名映射。

匿名映射

　　当进程需要创建一个较大的内存时，可以使用mmap()来创建匿名映射，使用匿名映射来分配内存的好处：

  • 不必关心内存碎片，例如，当进程不再需要使用匿名映射的内存，那么可以撤销内存映射，内存则会立即返回给系统。
  • 匿名映射的内存其大小是可以调整的。

　　当然，匿名映射适合用来分配较大的内存，否则就优点浪费了（杀鸡焉用牛刀？）：

  • 匿名映射的内存大小是以 page 作为单位的，如果分配的内存不是 page 大小的整数倍，则会浪费内存。
  • 创建匿名映射的开销比较大，如果分配的是小块内存，使用mmap()就不划算了。

Linux 进程编程（三）-- 进程调度

发表于 2015-07-28   |   分类于 系统编程

进程调度

　　进程调度器（Process Scheduler）将有限的处理器（Process）资源分配给多个进程，也就是说，进程调度器决定了特定的进程什么时候执行并且执行多长的时间。

　　多任务的操作系统有两种：

  • 在协作式（Cooperative）的多任务系统中，进程调度的行为由进程自己来决定，也就是说，进程将会一直执行，直到进程主动让出（Yielding）处理器。设想一下，要是进程长时间执行而不主动让出处理器，那么，其它进程将得不到执行的机会。
  • 在抢占式（Preemptive）中，进程调度的行为将取决于调度器，调度器则会让特定的进程执行一段时间，之后再执行另一进程。

　　Linux 是抢占式的多任务系统，在传统的 Linux 进程调度中，每个进程什么时候执行并且执行多长时间，由两个变量来决定，priority 和 timeslice，priority 决定了进程的优先级，timeslice 则指定了进程在被抢占之前能运行多长时间，要是进程耗尽了 timeslice，系统可能会开始执行另一进程。

　　但是从 Linux 2.6 开始使用的调度器称为完全公平调度器（Completely Fair Scheduler），即 CFS。

进程优先级

　　进程的优先级将会影响进程什么时候执行，还能影响进程的执行时间。我们可以使用nice()来设定进程的 nice 值，以此来影响进程的优先级。nice 的取值范围为[-20, 19]，默认值是 0，当我们将 nice 值设低时，进程的优先级则会提高，同样道理，更高的 nice 值则表示进程的优先级将降低。

　　调用nice()可以用来增加 nice 值，当然，要是inc的值为负，则表示降低 nice 值，nice()将会返回更新后的 nice 值。我们知道，降低 nice 值则表示提升进程的优先级，这可能需要 root 权限。

编译与链接（二）-- 链接

发表于 2015-07-25   |   分类于 体系结构

链接过程

　　当编译器遇到未定义的函数和全局变量时，会假定它们存在于其它文件中，那么，链接器就是确定这些未定义的函数和全局变量究竟定义在哪些地方，并使未定义符号的引用到它们定义的地方。

　　链接器主要执行两个任务：

  • 符号决议：使多个目标文件中的符号引用关联到唯一的符号定义。
  • 重定位：编译器和汇编器负责产生可重定位的目标文件。链接器对这些目标文件的相同区段 ( section ) 进行合并，并为每个符号定义确定内存地址，同时使符号引用指向它们定义的位置。

符号决议

　　在链接的过程，要是链接器无法找到符号定义在哪个地方，则会报错：
　　

　　另一方面，符号可能出现多次定义：

  • C++ 中规定符号仅能定义一次。
  • C 则规定函数和初始化的全局变量仅能定义一次，但未初始化的全局变量则可以定义多次。

　　在 C 语言中，函数和初始化的全局变量为强 ( Strong ) 符号，而未初始化的全局变量为弱 ( Weak ) 符号。
链接器符号决议的规则：

  • 强符号最多有一个。
  • 一个强符号和多个弱符号，则选择强符号作为符号定义。
  • 多个弱符号，则选其中一个用作符号定义。

  • 当多个进程需要使用同一共享库，共享库的 .text 区段会被载入到内存中，供这些进程共享。

epoll instance

　　在了解 epoll 编程之前，需要先了解什么是 epoll instance ？

　　我们知道，在事件驱动的网络编程中，我们需要对感兴趣的文件描述符进行监视，那么可以将这些文件描述符合称为 interest list，另一方面，我们还需要知道在这个 interest list 中，哪些成员处于 ready for I/O 的状态，并将这些成员合称为 ready list。

　　epoll instance 实际上是用来记录 interest list 以及 ready list，这样就可以通过这个 epoll instance 来了解这些文件描述符的状况。

epoll API

　　epoll_create()用来创建一个 epoll instance，并返回一个文件描述符，用来引用这个 epoll instance。当epoll_create()返回时，epoll instance 中的 interest list 是空的。

　　这样我们知道，对于 non-blocking 的 socket 来说，read()返回 -1时，这时需要检查errno是否为EWOULDBLOCK或EAGAIN，若是，则说明已经读完了所有的数据，write()也是同样道理。

Edge-trigger

　　将文件描述符添加进 epoll instance 的 interest list 中时，可以指定事件的触发方式，即 level-triggered 模式和 edge-triggered 模式，默认情况下为 level-triggered 模式。

　　好了，那这两种模式的区别？

  • 在 level-triggered 模式下，只要某个 socket 是可读 ( readable ) 或可写的 ( writeable )，那么调用epoll_wait()会将其返回在 ready 事件中。这时要是我们没有将 socket 中的数据完全读取 ( 留下一部分未读取 )，那么，下一次调用epoll_wait()时，同样会将这个 socket 以 ready 事件的形式返回，因为这个 socket 仍然是可读的 ( readable )。
  • 而在 edge-triggered 模式下，当某个 socket 从 unreadable 转变成 readable 或是从 unwriteable 转变成 writeable 的，那么epoll_wait()会将其返回在 ready 事件中。这时要是我们没有将 socket 中的数据完全读取 ( 留下一部分未读取 )，那么，下一次调用epoll_wait()时，这个事件就不会再返回了。

　　使用 edge-triggered 模式时，需要将 socket 设置成为非阻塞的。既然 edge-triggered 模式只会将事件返回一次，则应当一次性将 socket 中的数据全部读取出来。

　　另一个问题是，在 edge-triggered 模式下，当epoll_wait()返回时，这时可能有多个 client 的连接请求，但 epoll 却只会触发一次事件，因此，我们需要在循环中调用accept()来接受多个 client 的连接请求，当accept()返回-1并且将errno设置成EAGAIN或EWOULDBLOCK时，那就表示已经处理完所有的连接了。

　　使用 edge-triggered 模式：

  • 首先将 epoll_instance 监视的所有文件描述符都设置成非阻塞的。
  • 构建 epoll_instance 的 interest list。
  • 通过 epoll_wait()返回 epoll instance 的 ready list。
  • 历遍这个 ready list，对其中的文件描述符不断执行 I/O 系统调用，直到这些系统调用返回 -1 并将errno设置成EAGAIN或EWOULDBLOCK。
