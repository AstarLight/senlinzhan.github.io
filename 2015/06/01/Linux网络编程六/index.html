Linux 网络编程（一）-- 理解 TCP 协议

发表于 2015-06-01   |   分类于 网络编程

TCP Segment 格式

　　首先，让我们了解 TCP segement 的格式：

　　注意到，TCP segment 不包含 IP 地址，但包含源端口号和目的端口号。TCP 字节流中每个字节都有一个序号，序号用来表示字节在字节流中的位置，Sequence number 则是这个 segment 中数据的第一个字节的序号。既然每个 segment 都有一个序号，那么接收端就可以根据 segment 的序号对这些 segment 进行重新组合。

　　Acknowledgement Number 即 ACK，用来确认收到 segment，当接收端收到一个 segment 之后，会返回一个 ACK，其值为接收到的 segment 最后一个字节的序号加一。

　　Window size 为窗口大小，也即 Advertised window，当接收端发送 ACK 时，Window size 就表示接收端的缓冲区还有多少空间用来接收数据，从而确保发送端发送的数据不会使接收端的缓冲区溢出。

TCP 连接建立

　　TCP 建立连接时，通信双方需要向对方交换自己的初始序号，即 ISN (Inital Sequence Number)，这个过程被形象地称为三次握手：

  • Client 调用connect()主动建立连接，Client 会选择一个特定的序号作为初始序号，并发送一个 SYN segment 给 Server，目的是将 ISN 发送给 Server。
  • 接收到 Client 的 SYN segement 之后，Server 会将 ACK 连同自身的 ISN 作为一个 SYN segment 发送给 Client。
  • Client 发送 ACK segment 给 Server。

TCP 连接的终止

　　Client 会调用close()请求主动关闭连接。我们知道，TCP 连接是全双工的，数据可以在两个通道上传递，那么，关闭连接的时候则需要关闭两个方向，两个 FIN segment 加上两个 ACK，这样，TCP 连接的终止则需要经过 4 次握手：

  • Client 发送一个 FIN segment 给 Server。
  • Server 会发送 ACK 作为响应，在此之后，在 Server 调用read()都会返回 0。
  • Server 需要等待一段时间用于关闭连接，之后发送 FIN segment 给 Client。
  • 收到 Server 的 FIN segment 之后，Client 发送 ACK 作为响应。

　　当 Client 接收到 Server 的 FIN segment 之后，Client 此时处于 TIME_WAIT 状态，TIME_WAIT 状态有两个作用：

  • 使 Client 将会等待一段时间，这有助于可靠地终止连接，
  • 等待旧的重复的 segment 在网络中消失，这样当新连接建立时，就不会收到这些 segment 了。

　　TIME_WAIT 状态也即 2MSL 状态，TCP 的每个 segment 都有最大生存时间 (Maximum Segment Lifetime)，也即 MSL，Linux 设定 MSL 为 30 秒。
　　
　　为什么 TIME_WAIT 状态需要持续 2MSL 时间呢？好了，假定 Client 最后发送的 ACK 丢失了，为了保证可靠地关闭，Server 则需要重传 FIN，这时， Client 处于 TIME_WAIT 状态，由于这个状态将持续 2SML 时间，也就保证了 Client 在收到这个 FIN 之后能够再次发送 ACK。不过需要注意，Server 是被动关闭的一方，这样 Server 也就不需要 TIME_WAIT 状态。

TCP 连接的半关闭

　　TCP 连接是全双工的，也就是说既可以写，也可以读。当调用close()来关闭 socket 描述符时，则会关闭这个 socket 的写和读的功能，并会释放这个 socket 描述符。

　　但有时我们需要关闭连接的一个方向，例如，Client 不想发送数据，但仍然希望从 Server 那里读取数据，那么，可以调用shutdown()来关闭写入通道：

　　how参数可以是SHUT_WR(关闭写入通道)，SHUT_RD(关闭读取通道)，也可以是SHUT_RDWR(关闭读取通道和写入通道)。但通常来说SHUT_RD并不常用，由于不同版本的 Linux 对SHUT_RD的支持并不一致，因此，在可移植的 TCP 程序来说，应当避免使用SHUT_RD操作。

　　当 Client 调用shutdown()关闭写入通道时，Client 发送一个 FIN segment，Server 则发送相应的 ACK。在此之后，Client 仍可以读取数据，Server 仍可以发送数据。当 Server 调用close()关闭连接时，则会发送 FIN 给 Client，接着 Client 发送 ACK 给 Server。

　　注意到shutdown()并不会释放 socket 描述符，即使调用shutdown()关闭了连接，仍然需要调用close()来关闭 socket 描述符。

TCP 的超时与重传

　　我们知道，TCP 协议是可靠的，面向连接的，双向字节流的。TCP 保证可靠传输就是发送一个数据，并等待对方发送 ACK，用来确认对方收到数据。但不要忘了一点，ACK 也有可能丢失啊。

　　为应对 ACK 可能丢失的情况，TCP 使用一个 Timer，当 Timer 触发时，若还未收到 ACK，则会重新发送数据，这就是 TCP 的超时与重传机制。

TCP 的拥塞控制

　　TCP 的流量控制主要用来限定发送端数据不能发送过快，超过对方的接收能力，也就是说发送的数据不能使对方的缓冲区溢出。TCP segment 中有一个字段 Window size，也称为通知窗口，即 Advertised Window，接收端通过 Advertised Window 来告诉发送端，自己的缓冲区还有多少空间来接收数据，于是发送端会调整自己发送数据的快慢。

　　我们知道，TCP 流量控制是限制发送端不能发送太快，防止压垮接收端，但 TCP 的流量控制则无法应对网络负载过大的情况。在网络负载过大的时候，如果发送端发送的数据过快，而路由器来不及转发，数据就会丢失，这时发送端需要重传数据，同样，重传数据又将加速网络的拥塞程度，如此循环下去，则会严重压垮网络。

　　TCP 的拥塞控制则是防止发送端发送太快而压垮网络，发送会根据 segment 的丢失来检测是否发生了拥塞，若发生了丢失，则会认为发生了拥塞，这时发送端则会降低其数据传输速率。

　　最初的拥塞控制使用慢启动（Slow Start）和拥塞避免（Congestion Avoidance）算法。

　　TCP 的拥塞控制主要使用拥塞窗口（Congestion Window）来控制，拥塞窗口简称 cwnd。慢启动算法会使发送端一开始以较慢的速度发送 segment，随着接收端响应 ACK 之后，发送端的发送速率会以指数级增长。

　　慢启动算法：

  • 连接建立之后，cwnd 初始值为 1，表示一个 MSS 的大小
  • 每当收到一个ACK，cwnd = cwnd + 1
  • 每经过一个 RTT ( Round Trip Time )之后，cwnd = cwnd * 2
  • 当 cwnd >= ssthresh ( Slow Start Threshold )之后，就会使用拥塞避免算法，ssthresh 的值通常为 65536

　　当 cwnd >= ssthresh 时，TCP 会使用拥塞避免算法来平衡负载，这样就可以避免增长过快导致网络拥塞，慢慢的增加调整到网络的最佳值。拥塞避免算法：

  • 收到一个ACK时，cwnd = cwnd + (1 / cwnd)
  • 当每过一个RTT时，cwnd = cwnd + 1

　　TCP 根据是否丢包来检测是否发生了拥塞，这样当需要超时重传时，发生拥塞的可能性就比较大。TCP 应对拥塞状态的算法：

  • sshthresh = cwnd /2
  • cwnd 重置为 1
  • 进入慢启动过程
