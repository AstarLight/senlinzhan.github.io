发表于 2015-06-08   |   分类于 网络编程

TCP Server

　　典型的 TCP 的 Server-Client 模型：

　　Server 调用socket()得到 socket 描述符，而bind()则将这个 socket 绑定到本地主机特定的端口上，接着，调用listen()在这个端口对输入的连接请求进行监听。要是 Server 监听到 Client 的连接请求，那么，Server 将调用accept()来接受 Client 的连接请求。

　　每次调用accept()都会返回一个 socket 描述符，由于 TCP 连接是全双工的，因此我们可以利用这个 socket 描述符读取从 Client 发送来的信息，也可以向 Client 发送信息。

　　socket()返回的 socket 描述符可以用来引用这个新创建的 socket。

　　我们可以用bind()将 socket 绑定到本地主机的特定端口上：

　　通过addr参数，bind()可以为 Server 指定需要监听的 IP 地址以及端口号，当然，IP 地址必须是本地主机的合法 IP 地址，也就是说，我们不能提供其它主机的 IP 地址。另一方面，使用bind()时，要是指定的端口号小于 1024，那么，需要超级用户的所有权。

　　在为 Server 创建 socket 的时候，我们通过socket()的domain参数指定了使用 IPv4 协议或者是 IPv6 协议。要是我们创建的是使用 IPv4 协议的 socket，那么，为bind()提供的应当是 IPv4 格式的地址。

　　bind()返回的常见错误是 EADDRINUSE 错误，表示 Address already in use。

　　你可能觉得，在调用bind()的之前应该手动填充sockaddr_in结构体，再将这个结构体的指针传递给bind()。但实际上，我们不必这样做，因为可以调用getaddrinfo()来简化工作：

　　我们可以将host设置成NULL，同时将hint的ai_flags成员设置成AI_PASSIVE，这表示我们将查询本地主机的 IP 地址。getaddrinfo()将会返回一个链表，包含多个 IP 地址信息。对于 IPv4 的 IP 地址，getaddrinfo()会将s_addr设置成INADDR_ANY，而要是 IPv6 的 IP 地址，getaddrinfo()则会将sin6_addr设置成in6addr_any。由于这样的设置，那么接下来调用bind()的时候，bind()就会将本地主机的 IP 地址填充到sockaddr_in或sockaddr_in6结构体里面。

　　listen()系统调用：

　　我们知道，每次 Server 调用accept()的时候，Server 就能处理一个 Client 的连接，也就是说，为了处理多个 Client 的连接请求，Server 必须多次调用accept()。

　　在 Server 调用accept()之前，Client 可能会调用connect()请求连接到 Server，这种情况就称为 Pending Connection。listen()的backlog就是用来限制这种 Pending Connection 的个数的。

　　要是 Pending Connection 的数目超出了backlog的数值，那么多出来的那部分连接请求则会被系统拒绝。

　　调用accept()用来接受 Client 的连接请求，要是在调用accept()时没有 Pending Connection，那么这时accept()就会阻塞，等待 Client 的连接请求。

　　对于 Server 来说，有时需要知道 Client 的地址，那么 Client 的地址将通过addr来返回，但是需要注意一点，在调用accept()之前，addr需要指向一个结构体，因为这个结构体需要用来存储 Client 的地址，并且addrlen也需要初始化为这个结构体的大小。但在调用accept()之后，addrlen的值将被设置为 Client 的地址实际占用的大小。

　　当然，要是 Server 不想知道 Client 的地址，那么，可以将addr和addrlen分别设置为NULL和0。

TCP Client

　　TCP Client 使用connect()与 TCP Server 建立连接：

　　addr表示远程主机的地址，addrlen表示这个地址的大小。通常我们是调用getaddrinfo()来查询远程主机的 IP 地址，并且查询的结果传递给connect()即可。

　　要是connect()连接失败，那就需要进行重新连接，这时可以先调用close()来关闭 socket，并且重新创建新的 socket，根据这个新的 socket 重新进行连接。

　　TCP Client 在调用connect()不必调用bind()，因为内核会选择一个临时端口赋予 TCP Client。

　　TCP Client 调用connect()将激发 TCP 的三次握手：

　　调用connect()时，只有在连接建立，或者出现错误时，connect()才会返回，有几种错误的情况：

  • 在 Client 发送 SYN segment 之后，要是 Server 没有对此作出响应，则connect()将返回 ETIMEOUT，表示超时错误。
  • 在 Client 发送 SYN segment 之后，收到 RST segment 响应。
  • 若 Client 发送的 SYN segment 在某个路由器上引发 destination unreachable 的 IMCP 错误，则connect()将返回 EHOSTUNREACH 或 ENETUNREACH 错误。

　　当 Client 收到 Server 发送的 SYN segment 时，connect()才会返回，而 Server 则需要等到收到 Client 发送的 ACK segment 之后，accept()才会返回。

关闭连接

　　调用close()可以用来关闭 socket 描述符，并且使得与之关联的 file table entry 的引用计数减 1，当引用计数变成 0 之后，这个 socket 的资源才会被释放。

　　在多进程的并发服务器中，TCP 连接正常终止的步骤：

  • Client 可以调用close()显式关闭 socket descriptor，或者在进程终止之后，由进程会自动关闭 socket descriptor。这将导致 Client 发送一个 FIN segment 给 Server 子进程，Server 则会发送 ACK segment 作为响应。这个时候，Server 子进程将处于 CLOSE_WAIT 状态，而 Client 则处于 FIN_WAIT2 状态。
  • Server 子进程调用read()处于阻塞状态，当收到 Client 发送的 FIN segment 之后，read()将返回 0，这个时候，Server 子进程可以调用close()关闭 socket descriptor 或者直接退出进程。当 Server 子进程关闭 socket descriptor 之后，将发送一个 FIN segemtn 给 Client，Client 将发送 ACK segment 作为响应。这时，Client 将处于 TIME_WAIT 状态。

    数据传送

    　　对于 TCP socket 来说，我们使用send()来发送数据，同时使用recv()来接收输入的数据。

　　对于send()来说，buf指向需要发送的数据，而nbytes表示数据的字节长度，要是我们发送的是正常的数据，那就将flags的值设定为 0。注意到，实际发送的数据可能会小于nbytes指定的字节长度，要是发送成功，send()会返回实际发送的字节长度，发送失败则会返回 -1。
　　recv()的参数与send()的类似。注意到调用recv()的时候，要是没有数据可读取，那么recv()将会阻塞。
