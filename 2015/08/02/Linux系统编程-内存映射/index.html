Linux 系统编程 -- 内存映射

发表于 2015-08-02   |   分类于 系统编程

内存映射

　　进程可以将磁盘中的文件映射到其 virtual memory 中，这称为内存映射( Memory Mapping )。当文件被映射到 virtual memory 中，进程就可以访问内存的相应区域，以此来访问文件。

　　实际上，当文件映射到 virtual memory 时，文件并不存在于 physical memory 中。当进程需要访问文件时，相应的 page 才会从 virtual memory 换入到 physical memory 中。

　　另一方面，也可以将 virtual memory 的一个区域映射到匿名文件( Anonymous File )。当进程引用到这个区域的一个 virtual page 时，内核则会在 physical memory 中找到一个 victim page，如果这个 page 被修改过，则将这个 page 换出，接着将这个 page 的内容初始化为 0 并更新进程的 page table。

共享对象

　　我们知道，一个程序可以对应多个进程，那么，这些进程的会拥有相同的 .text 区域，另一方面，进程也会拥有相同的 shared library。假定每个进程都在 physical memory 中拥有这些相同的对象的一个副本，则无疑会占用过多的内存。

　　一个进程所映射的内存可以与其它进程共享，也就是说，同一个程序的多个实例会共享相同的 .text 区域，并且共享相同的 shared library，这些相同的对象在 physical memory 只存在一份副本。一个对象可以映射到多个进程的 virtual memory 的一个区域，可以分为共享映射和私有映射。

　　在共享映射的情况下，多个进程将一个对象映射到它们各自的 virtual memory 的一个区域，那么，要是有一个进程修改了这个对象，这个修改的操作对其它进程来说都是可见的。

　　私有映射则不同，私有映射采用了写时拷贝( Copy On Write )的技术。假定两个进程将一个对象映射到它们各自的 virtual memory 的一个区域：

fork 的语义

　　我们知道，调用fork()则会创建一个与父进程几乎一致的子进程，设想一下，在调用fork()时，要是系统完整地拷贝（即深度拷贝）父进程，那这代价也太大了。

　　实际上，调用fork()时，系统会为子进程创建必要的数据结构（ page table 或 file descriptor table 等等），子进程的 virtual memory 和父进程的 virtual memory 是一致的，当父进程或者子进程需要对共享区域进行修改时，则会使用 Copy On Write 的技术，创建需要修改的 page 的副本，并对这个副本执行修改操作。

创建映射

　　mmap()请求内核将文件的一个区域映射到进程的 virtual memory 的一个区域，通常将addr设置成NULL。

　　prot用来指定映射区域的访问权限：PROT_NONE 表示无法访问这个区域，PROT_READ 表示这个区域的 page 可读，PROT_WRITE 表示这个区域的 page 可写，PROT_EXEC 表示这个区域的 page 可被执行。

　　而flags则可以用来指定创建的是共享映射还是私有映射，是文件映射还是匿名映射：MAP_PRIVATE 表示创建的是私有映射，MAP_SHARED 表示创建的是共有映射，MAP_ANONYMOUS 表示创建的是匿名映射。

匿名映射

　　当进程需要创建一个较大的内存时，可以使用mmap()来创建匿名映射，使用匿名映射来分配内存的好处：

  • 不必关心内存碎片，例如，当进程不再需要使用匿名映射的内存，那么可以撤销内存映射，内存则会立即返回给系统。
  • 匿名映射的内存其大小是可以调整的。

　　当然，匿名映射适合用来分配较大的内存，否则就优点浪费了（杀鸡焉用牛刀？）：

  • 匿名映射的内存大小是以 page 作为单位的，如果分配的内存不是 page 大小的整数倍，则会浪费内存。
  • 创建匿名映射的开销比较大，如果分配的是小块内存，使用mmap()就不划算了。

　　malloc()的内部实现策略是，如果分配的内存小于一个阈值，那么就在 heap 中分配内存，要是分配的内存超过这个阈值，则使用mmap()来分配匿名映射的内存，这个阈值通常是 128KB。
