信号的简介

　　通常，进程和内核可以通过信号 ( Signal ) 来终结其它进程，停止其它进程的执行，或者仅是通知其它进程。例如，要是进程执行非法的浮点数运算，内核则会发送SIGFPE信号。另一方面，要是子进程终止或停止执行了，则内核发送SIGCHLD信号给其父进程。常见的信号：

　　每种信号都有默认的处理方式，除了SIGSTOP和SIGKILL信号之外，我们可以为其它信号显式定义处理方式。

　　若handler为SIG_IGN，则表示忽略处理信号，要是SIG_DFL则表示恢复为默认的信号处理方式。通常，handler为函数指针，指向自定义的信号处理函数。

　　要是信号已经发送但还未被接收，则称为 pending 信号。在同一时刻，进程的所有 pending 信号中，同一类型的信号最多只能出现一次。处理信号过程中可能出现的问题：

　　由于第二个问题，也就是说相同类型的信号可能会丢失，因此，我们不能通过接收到的信号的次数来判定事件出现的次数。

　　在 Web Server 中，通常需要创建多个子进程，当子进程终止时，父进程需要负责清除子进程，否则，这些已经终止的子进程将会继续占用系统资源。当子进程终止时，内核会发送SIGCHLD信号给父进程，父进程可以通过自定义的信号处理方式来回收子进程。

　　在清理子进程的时候，需要注意上面提及的问题：

  • 当第一个子进程终止时，内核会发送SIGCHLD信号给父进程，父进程调用信号处理器，清除这个子进程，在清除子进程的过程中，下一个SIGCHLD信号将会阻塞在队列中，并且由于信号队列中最多只能存在一个SIGCHLD信号，意味着接下来要是内核继续发送给父进程SIGCHLD信号，则会丢失。解决这个问题，可以在信号处理器中尽可能多地清除子进程。
  • 另一方面，父进程调用信号处理器时，系统调用可能会被中断，这时需要手动重启系统调用。
