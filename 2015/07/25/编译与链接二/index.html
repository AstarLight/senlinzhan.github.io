编译与链接（二）-- 链接

发表于 2015-07-25   |   分类于 体系结构

链接过程

　　当编译器遇到未定义的函数和全局变量时，会假定它们存在于其它文件中，那么，链接器就是确定这些未定义的函数和全局变量究竟定义在哪些地方，并使未定义符号的引用到它们定义的地方。

　　链接器主要执行两个任务：

  • 符号决议：使多个目标文件中的符号引用关联到唯一的符号定义。
  • 重定位：编译器和汇编器负责产生可重定位的目标文件。链接器对这些目标文件的相同区段 ( section ) 进行合并，并为每个符号定义确定内存地址，同时使符号引用指向它们定义的位置。

　　另一方面，符号可能出现多次定义：

  • C++ 中规定符号仅能定义一次。
  • C 则规定函数和初始化的全局变量仅能定义一次，但未初始化的全局变量则可以定义多次。

　　在 C 语言中，函数和初始化的全局变量为强 ( Strong ) 符号，而未初始化的全局变量为弱 ( Weak ) 符号。
链接器符号决议的规则：

  • 强符号最多有一个。
  • 一个强符号和多个弱符号，则选择强符号作为符号定义。
  • 多个弱符号，则选其中一个用作符号定义。

静态链接

　　要是多个程序需要引用相同的目标文件，那么，可以考虑将这些目标文件打包成为静态库。静态库里面包含多个目标文件，但是，我们的程序可能只依赖其中一部分目标文件，因此，在静态链接的过程中，链接器只会将需要用到的目标文件链接到可执行程序中。经过静态链接而成的可执行文件，在执行时不再需要依赖其它的目标文件。

　　使用静态库带来的问题：

  • 目标文件嵌入到可执行文件中，这意味着无法替换目标文件。例如，若需要使用新版本的 print.o，则需要重新链接。
  • 程序通常会使用通用的库 ( 例如 C 标准库 )，在静态链接的情况下，每个可执行文件都会包含这个的库的副本，这会占据不必要的磁盘空间。另一方面，当操作系统运行多个进程时，每个进程的 text segment 会包含一些相同的代码 ( 例如printf()或scanf() )，这无疑会消耗内存。

动态链接

　　共享库为 .so 文件，共享库用来解决静态库带来的问题：

  • 一个共享库可供多个可执行文件共享，因此系统中只需维持一个共享库的副本。
  • 当多个进程需要使用同一共享库，共享库的 .text 区段会被载入到内存中，供这些进程共享。

加载程序

　　在可执行文件中，.bss 区段包含未初始化的全局变量，当可执行文件存储在磁盘中时，实际上没有必要为 .bss 区段分配空间，相反，可执行文件中仅仅记录 .bss 区段的地址以及大小。加载器会假定未初始化的全局变量的初始值为 0，这样进程的 bss segment 会默认初始化为 0。

　　初始化的全局变量则存储在 .data 区段，.data 区段包含这些全局变量的初始值，在加载的过程中，加载器会将全局变量的初始值拷贝到进程的 data segment。
