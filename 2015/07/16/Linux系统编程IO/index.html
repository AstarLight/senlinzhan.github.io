文件共享

　　我们知道，每个进程都拥有一个 file descriptor table，用来记录进程所打开的文件描述符，当进程调用fork()之后，子进程将得到父进程的 file descriptor table 的一份拷贝。

　　为了记录打开的文件，内核维护三个数据结构：

  • 每个进程都拥有其各自的 file descriptor table。
  • 系统级的 file table，用来记录系统打开的文件，所有进程共享这个 file table，table 的每个 entry 包含了当前文件的位置，引用计数，以及一个指向 v-node table entry 的指针。引用计数表示当前有多少进程引用到这个文件，当引用计数的值变成 0 时，系统则会删除这个 file table entry。
  • 文件系统的 v-node table，供所有进程共享。

　　假定进程打开了两个文件，descriptor table 中存在两个文件描述符：

　　接着，进程调用fork()创建了子进程，子进程将会得到父进程 descriptor table 的一份拷贝：

　　这时，父进程和子进程共享同一个 file table，也就是说，父进程和子进程共享相同的文件位置。另一方面，可以看到，文件的引用计数是 2，则表示只有父进程和子进程都关闭了文件，系统才会删除相应的 file table entry。

　　考虑另一种情况，要是进程两次调用open()打开同一个文件，那么，情况又是怎样的呢？

　　可以看到，存在两个文件描述符，每个文件描述符都有其各自的 file table entry，我们知道，file table entry 中包含文件位置和引用计数，也就说，我们可以利用这两个文件描述符，从同一个文件的不同位置读取数据。
